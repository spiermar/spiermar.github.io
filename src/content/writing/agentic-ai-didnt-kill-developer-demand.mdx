A lot of people thought agentic AI would kill developer jobs. The logic was simple: if it takes much less effort to build software, you need fewer people to build it. Makes sense on paper. But that doesn’t seem to be what’s happening.

What happened instead is that building got cheaper, so people started building more. More products, more experiments, more internal tools, more side projects, more random ideas that would have never made it past the “not worth the effort” stage before. That’s the part many people missed. Cheaper building does not automatically reduce demand for developers. It can increase demand, because it increases the amount of software people want to create. The broader point from Citadel’s piece is useful here too: capability growth is not the same thing as instant adoption or replacement. The real world still has friction. Orgs move slowly. Trust is limited. Deployment is messy. ([citadelsecurities.com](https://www.citadelsecurities.com/news-and-insights/2026-global-intelligence-crisis/))

But I think there’s another shift that matters just as much. Development now feels radically better. Not just faster, better. Before, building meant constant context switching: write code, run tests, read docs, search errors, debug, try again, break flow, repeat. Now a lot of that collapses into one session, one stream, one loop of instructions, feedback, and iteration. It’s smooth, and that smoothness changes behavior.

I’ve noticed it on myself multiple times. I start building new features or whole little projects for no serious reason other than I feel like it. I want to try something, see it work, follow the thread. Some of those things get used once. Some get abandoned a few days later. Some probably should not have existed at all. In the past, the higher development bar forced prioritization. You had to choose more carefully. You had to think harder about whether something was worth doing. The effort was a filter. Now that filter is much weaker.

You can get up and running with very little effort, and you can parallelize multiple projects at once. For an ADHD brain like mine, this is basically paradise. And honestly, it feels great. I can even see it on my Garmin watch. When I’m working with AI coding agents, my stress levels drop to almost zero for hours. It puts me into flow state much more easily than old-school development ever did. Less friction, less context switching, less resistance. That part is real.

But this is where the problem starts. Fast building creates the illusion of progress. A lot can be built quickly without improving anything. You can ship features faster and still make the product worse. You can generate code, flows, agents, tools, and experiments at insane speed and mostly create bloat. More output does not mean more value. That’s also why the Adaline post gets something important right: speed is not about shipping more, it’s about learning faster. Shipping velocity is not the point. Learning velocity is. Are you getting closer to something users actually want? Are you clarifying what good looks like? Or are you just producing more stuff because now you can? ([labs.adaline.ai](https://labs.adaline.ai/p/shipping-fast-and-iterating-at-ai))

That distinction matters a lot. Because if agentic AI makes software dramatically easier to produce, then we are going to produce a lot more software than deserves to exist. Some of it will be useful. A lot of it will be noise. Some of it will be abandoned. Some of it will keep running anyway. And once it’s running, it becomes someone’s problem. It needs maintenance, monitoring, debugging, cleanup, migration, ownership.

So no, I don’t think agentic AI is simply reducing the need for developers. I think it’s shifting where the value is. The scarce skill is becoming less about writing code from scratch and more about judgment: what should exist, what should not, what is actually improving the product, what is just motion disguised as progress, what should be killed early, and what someone will have to maintain later.

Agentic AI removed a lot of the friction that used to force better decisions. That’s amazing for execution. It’s not great for restraint. And if we keep building faster than we can prioritize, evaluate, and clean up, then we may end up with the opposite of the original prediction: not fewer developers, but more demand for developers to manage the growing pile of AI-assisted software we keep creating.

A never-ending snowball.
